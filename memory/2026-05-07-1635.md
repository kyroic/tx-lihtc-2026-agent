## Current Extraction Architecture

The pipeline is a 4-step chain with clean separation:

```
run_pipeline.py
  → extract_one_pdf_final()
    → V5.8 Strategy (v5_8_fast_tiebreaker.py)
      → find_tiebreaker_pages_fast()    [pdftotext CLI — finds tie-breaker section]
      → extract_only_tiebreaker_pages() [pdfplumber — reads only 2 zones]
      → LLM call via chat_completions() [structured JSON output]
    → auto-recovery (census_tract, poverty_rank)
    → write_outputs() (CSV/XLSX/JSONL)
```

### The Key Design Decision: V5.8 Only Reads Two Zones

V5.8 doesn't read the entire PDF. It reads **only**:

| Zone | Pages | Content |
|---|---|---|
| **Standard pages** | First 10 | App name, contacts, census tract hints |
| **Tie-breaker pages** | Pages found by pdftotext | Full text + extracted tables |

Everything else is skipped. This is why it's fast — 13 minutes for 114 PDFs instead of 40+ minutes reading every page.

### How V5.8 Finds the Tie-Breaker Section

1. Runs `pdftotext -layout` on the full PDF (streaming, fast)
2. Splits by form-feed (`\f`) to get per-page text
3. Searches for exact headers: `"Tie-Breaker Information (Competitive HTC Only)"`, `"Tie-Breaker Information"`
4. Returns the page numbers where matches are found

### What V5.8 Sends to the LLM

The LLM receives **two data blocks** in the prompt:

1. `standard_pages` — first 10 pages, text only, 4000 chars each
2. `tiebreaker_pages_extracted` — the tie-breaker pages with:
   - `text` — full text (8000 chars)
   - `tables` — pdfplumber-extracted table data
   - `full_text` — complete un-truncated text

### What V5.8 Already Recovers from Tie-Breaker Pages

Right now the `schema_hint` only asks for names:

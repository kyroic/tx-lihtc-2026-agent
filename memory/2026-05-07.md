## Final Architecture (6 Layers)

1. **DISCOVERY** (`discover.py`) — Python web crawler, collects PDF URLs
2. **DOWNLOAD** (`download.py`) — Parallel download with manifest tracking
3. **EXTRACTION** (`v5_8_fast_tiebreaker.py`) — Core LLM extraction
4. **AUTO-RECOVERY** (`run_v5_9b_census_recovery.py`, `run_v5_9c_poverty_recovery.py`) — Regex fallbacks
5. **AGGREGATION** (`extract.py`) — Write CSV/Excel/JSONL outputs
6. **ORCHESTRATION** (`run_pipeline.py`) — Single entry point

## Phase 2: Tie-Breaker Detail Extraction (IN PROGRESS — May 7, 2026 16:30 EDT)

### Client Request (from Sang)
The client needs additional fields extracted from the tie-breaker section of each PDF:

**Site coordinates:**
- `site_lat` — Site latitude
- `site_lng` — Site longitude

**Per-amenity details (4 amenities × 4 fields each = 16 new fields):**
- `distance_to_park`, `park_lat`, `park_lng`
- `distance_to_school`, `school_lat`, `school_lng`
- `distance_to_grocery`, `grocery_lat`, `grocery_lng`
- `distance_to_library`, `library_lat`, `library_lng`

**Aggregate score:**
- `tiebreaker_score` — Overall tie-breaker score

**Key constraint:** All this data lives in the same tie-breaker section we already extract. The V5.8 strategy already isolates and sends these pages to the LLM — we just need to ask for 19 more fields from the same input.

### Implementation Approach
- **Format:** Same wide format as existing CSV (one row per application, all columns inline)
- **Strategy:** Extend V5.8's existing `schema_hint` dict with 19 new fields — no structural changes needed
- **Files to touch:** `extract.py` (data model + output), `v5_8_fast_tiebreaker.py` (extraction schema), `run_complete_pipeline.py` (optional auto-recovery), `README.md` (docs)
- **What does NOT change:** `discover.py`, `download.py`, `model_client.py`, two-zone page extraction, the 11 existing fields
- **Risk:** Near zero — additive changes only, existing extraction path untouched

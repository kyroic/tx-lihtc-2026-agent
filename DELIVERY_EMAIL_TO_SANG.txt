================================================================================
PROJECT JOURNEY: V1 → V5.8 (CURRENT)
================================================================================

V1-V2: OpenClaw Browser Automation
-----------------------------------
Approach: Used OpenClaw to automate browser interactions with TDHCA website
Issues discovered:
  - Browser automation added 3-5x overhead vs direct API calls
  - Session management was fragile — frequent timeouts on large PDFs
  - Memory crashes on PDFs >200MB (some applications are 400+ pages, 377MB)
  - Each orchestration decision point added 30-60 seconds of latency
Result: Too slow for production, unreliable on large documents

V3: OpenClaw Orchestration + Classification
--------------------------------------------
Approach: Added intelligent classification layer via OpenClaw
Improvements: Better structure, could identify full applications vs supporting docs
Issues remaining:
  - Still dependent on OpenClaw runtime/gateway
  - Extraction speed limited by browser proxy layer
  - Complex multi-agent coordination added failure points
Result: Better accuracy, but still not production-viable

V4: Standalone Python Rebuild (Turning Point)
---------------------------------------------
Decision: Rebuilt entire pipeline as standalone Python agent
Changes:
  - Direct LLM API calls (no browser proxy)
  - Local file I/O (no gateway dependency)
  - Streaming PDF text extraction with pdftotext CLI
  - Parallel processing with threading
Results:
  - 3x faster than V3
  - Stable on 400+ page PDFs
  - No external runtime dependencies
Lesson: OpenClaw was adding complexity without proportional value

V5.0-V5.4: Targeted Tie-Breaker Discovery
------------------------------------------
Breakthrough: Discovered 100% of PDFs contain Tie-Breaker pages
Problem solved: Earlier versions only searched first 50 pages, but Tie-Breaker
                data is scattered throughout (pages 100-400+)
Approach: Search ALL pages for "Tie-Breaker" keyword, extract those pages only
Result: Found Tie-Breaker pages in all 114 PDFs (previously missed ~70%)

V5.5: Chunked Extraction (Memory-Safe)
--------------------------------------
Approach: Process PDFs in manageable chunks instead of loading entire document
Result: Zero OOM crashes, can handle 377MB PDFs reliably

V5.6: AI Folder Selection
--------------------------
Approach: AI determines which discovered folder contains full applications
Result: No manual folder specification needed

V5.7: Broad Heading Search
---------------------------
Approach: Search for variant Tie-Breaker page titles
          ("Tie-Breakers", "TIE-BREAKER", "Tie Breaker Information", etc.)
Result: Handles all heading format variations

V5.8 (CURRENT): Fast Extraction + Auto-Recovery
------------------------------------------------
Approach: pdftotext for speed + regex recovery for missing fields
Features:
  - census_tract: Regex scan all pages for 11-digit GEOID
  - poverty_rank: Search near "Quartile:" field (multiline patterns)
  - tiebreaker_*: Broad heading search + amenity keyword proximity
Result: 100% census tract, 97%+ poverty rate, 93% tie-breaker fields

================================================================================
WHY WE MOVED AWAY FROM OPENCLAW
================================================================================

Performance comparison (measured):

| Metric                  | OpenClaw (V1-V3) | Standalone (V4+) |
|-------------------------|------------------|------------------|
| Avg extraction time/PDF | 120-180 seconds  | 40 seconds       |
| Max PDF size handled    | ~200MB           | 400+MB (377MB+)  |
| Crash rate              | ~30%             | 0%               |
| External dependencies   | OpenClaw gateway | None             |
| Setup complexity        | High             | 5-minute setup   |

Key learnings:
1. Browser automation was overkill for this use case
2. Direct API + local processing was faster and more reliable
3. OpenClaw's session management added latency without proportional value
4. Simpler architecture = easier debugging and maintenance

================================================================================
FINAL SOLUTION: CURRENT ARCHITECTURE
================================================================================

High-Level Flow:
----------------
TDHCA Website → [Discover] → [Download] → [Extract] → [Recovery] → [Output]

Layer-by-Layer Breakdown:
--------------------------

LAYER 1: DISCOVERY
File: lihtc_tx_2026_agent/discover.py
Purpose: Crawl TDHCA website, collect PDF URLs
Technology: Python urllib + HTML parsing (no browser automation)
Process:
  1. Start from seed URL (TDHCA competitive 9% HTC page)
  2. Follow in-domain links up to max_pages (default 550)
  3. Collect .pdf URLs matching regex (default includes "2026")
  4. Filter to target folder (default: 2026-9-challenges)
Output: discover_state.json (list of PDF URLs)

LAYER 2: DOWNLOAD
File: lihtc_tx_2026_agent/download.py
Purpose: Download PDFs with retry logic and manifest tracking
Technology: urllib, threading, SHA-256 verification
Process:
  1. Check existing files (skip if already downloaded)
  2. Download in parallel (default 10 workers)
  3. Verify PDF header (%PDF magic bytes)
  4. Enforce size limits (default 500MB max per PDF)
  5. Track in download_manifest.json
Output: downloads/ folder with PDFs + manifest

LAYER 3: EXTRACTION (CORE)
File: lihtc_tx_2026_agent/strategies/v5_8_fast_tiebreaker.py
Purpose: Extract structured data from each PDF
Technology: pdftotext CLI, pdfplumber, LLM (GPT-4o-mini)
Process:
  1. Use pdftotext to extract text with page markers (fast)
  2. Search for Tie-Breaker page titles (broad pattern matching)
  3. Extract first 10 pages + Tie-Breaker pages only (not all 400+)
  4. Send targeted content to LLM with field schema
  5. LLM returns JSON with values + page numbers + quotes
Output: One row per PDF with 12 fields

LAYER 4: AUTO-RECOVERY
Files: run_v5_9b_census_recovery.py, run_v5_9c_poverty_recovery.py
Purpose: Fill missing fields that LLM extraction missed
Technology: Regex patterns, context-aware search
Process:
  census_tract:
    - Scan all pages for 11-digit GEOID pattern (48xxxxxxxxx)
    - Search for "Census Tract:", "GEOID:", "Tract:" labels
    - Validate format, take first match
  poverty_rank:
    - Search near "Quartile:" field (same line or next line)
    - Patterns: "Poverty Rate: XX.XX", "Poverty Rate\n  XX.XX"
    - Validate range (0-100%)
Output: Updated rows with recovered fields

LAYER 5: AGGREGATION & OUTPUT
File: lihtc_tx_2026_agent/extract.py
Purpose: Write final outputs in multiple formats
Technology: pandas, openpyxl, csv module
Process:
  1. Collect all extracted rows
  2. Deduplicate by PDF SHA-256 hash
  3. Write applications.csv (Excel-compatible)
  4. Write applications.xlsx (formatted Excel)
  5. Write applications.jsonl (rich format with evidence)
  6. Write review_queue.csv (rows needing manual check)
  7. Write run_summary.json (pipeline metrics)
Output: out/aggregate/ folder with all formats

LAYER 6: ORCHESTRATION
File: run_pipeline.py (root entry point)
Purpose: Single command to run entire pipeline
Technology: Python argparse, concurrent.futures
Process:
  1. Parse CLI arguments
  2. Run discovery (unless --skip-discover)
  3. Run download
  4. Run extraction with parallel workers (default 4)
  5. Run auto-recovery passes
  6. Write outputs
Output: Console progress + final summary JSON
